# 인수조건(AC) 패턴 가이드

> plan-req 스킬 참조용. SKILL.md §AC 작성 규칙을 보완하는 EARS 형식 패턴 + Good/Bad 비교.

---

## 1. EARS 형식 AC 패턴

### Ubiquitous (항상 성립)

```
[조건 없음] 시스템은 {행동}해야 한다.
```

| Good | Bad |
|------|-----|
| 시스템은 모든 사용자 입력을 UTF-8로 저장해야 한다 | 시스템은 적절한 인코딩으로 저장해야 한다 |
| 시스템은 모든 API 응답에 `Content-Type: application/json` 헤더를 포함해야 한다 | 시스템은 올바른 헤더를 반환해야 한다 |

### Event-driven (이벤트 발생 시)

```
{이벤트} 발생 시, 시스템은 {행동}해야 한다.
```

| Good | Bad |
|------|-----|
| 사용자가 "결제하기" 버튼 클릭 시, 시스템은 3초 이내에 PG사 결제창을 표시해야 한다 | 결제 버튼 클릭 시 결제가 빠르게 진행되어야 한다 |
| 세션 만료(30분 미활동) 시, 시스템은 로그인 페이지로 리디렉션하고 "세션이 만료되었습니다" 메시지를 표시해야 한다 | 세션 만료 시 적절히 처리해야 한다 |

### State-driven (특정 상태일 때)

```
{상태}인 동안, 시스템은 {행동}해야 한다.
```

| Good | Bad |
|------|-----|
| 결제 처리 중인 동안, "결제하기" 버튼은 비활성화(disabled) 상태를 유지해야 한다 | 결제 중에는 중복 결제를 방지해야 한다 |
| 오프라인 상태인 동안, 시스템은 로컬 캐시 데이터를 표시하고 "오프라인" 배지를 노출해야 한다 | 오프라인에서도 가능한 만큼 동작해야 한다 |

### Unwanted (원치 않는 상황)

```
{원치 않는 상황}이 발생하면, 시스템은 {대응}해야 한다.
```

| Good | Bad |
|------|-----|
| PG사 응답 타임아웃(10초 초과) 시, 시스템은 "결제 처리 중 오류가 발생했습니다. 잠시 후 다시 시도해주세요." 메시지를 표시하고 결제 상태를 "실패"로 기록해야 한다 | 결제 오류 시 사용자에게 알려야 한다 |
| 파일 업로드 용량 초과(10MB 초과) 시, 시스템은 업로드를 차단하고 "파일 크기는 10MB 이하만 허용됩니다" 메시지를 즉시 표시해야 한다 | 너무 큰 파일은 업로드되지 않아야 한다 |

### Optional (선택적 기능)

```
{조건/설정} 시, 시스템은 {행동}해야 한다.
```

| Good | Bad |
|------|-----|
| 관리자가 "다국어 지원"을 활성화한 경우, 시스템은 GNB에 언어 선택 드롭다운(한/영/중/일)을 표시해야 한다 | 필요에 따라 다국어를 지원해야 한다 |

## 2. 금지 표현 교정 패턴

> SKILL.md V2 금지 표현 7개를 교정하는 실무 패턴.

### "적절히" → 정량 기준

| Before (금지) | After (교정) |
|-------------|------------|
| 에러 메시지를 적절히 표시한다 | 에러 발생 시 2초 이내에 에러 코드 + 사용자 안내 메시지를 토스트로 3초간 표시한다 |
| 적절한 간격으로 데이터를 갱신한다 | 30초 간격으로 대시보드 데이터를 자동 갱신한다 |

### "필요에 따라" → 조건 명시

| Before (금지) | After (교정) |
|-------------|------------|
| 필요에 따라 캐시를 무효화한다 | 콘텐츠 수정 저장 시 해당 페이지의 CDN 캐시를 즉시 퍼지한다 |
| 필요에 따라 알림을 발송한다 | 주문 상태가 "배송중"으로 변경될 때 구매자에게 SMS+카카오 알림을 발송한다 |

### "가능한" → 수치

| Before (금지) | After (교정) |
|-------------|------------|
| 가능한 빠르게 로딩한다 | FCP(First Contentful Paint) 1.5초 이내, LCP 2.5초 이내로 로딩한다 |
| 가능한 많은 브라우저를 지원한다 | Chrome 90+, Safari 15+, Firefox 95+, Edge 90+를 지원한다 |

### "효과적으로" → 구체적 행동

| Before (금지) | After (교정) |
|-------------|------------|
| 검색 결과를 효과적으로 표시한다 | 검색 결과를 관련도 순으로 정렬하고, 검색어를 하이라이트하며, 페이지당 20건씩 표시한다 |

### "충분히" → 수치

| Before (금지) | After (교정) |
|-------------|------------|
| 충분한 보안을 제공한다 | 비밀번호는 bcrypt(cost 12)로 해싱하고, 로그인 5회 실패 시 15분 잠금한다 |

### "합리적" → 기준

| Before (금지) | After (교정) |
|-------------|------------|
| 합리적인 시간 내에 응답한다 | API 응답 시간 p95 기준 500ms 이내로 응답한다 |

### "적시에" → 시점

| Before (금지) | After (교정) |
|-------------|------------|
| 적시에 데이터를 백업한다 | 매일 03:00 KST에 전체 DB 스냅샷을 생성하고, 변경분은 1시간 간격으로 증분 백업한다 |

## 3. AC 구조화 패턴

### Given-When-Then (GWT)

복잡한 시나리오에 적합. 테스트 케이스와 1:1 매핑 가능.

```
Given: 사용자가 장바구니에 3개 상품을 담은 상태
When:  "전체 삭제" 버튼 클릭
Then:  장바구니가 비워지고, "장바구니가 비었습니다" 메시지 + 쇼핑 계속하기 CTA 표시
```

### 체크리스트형

단순 확인 항목에 적합. FR당 3~7개.

```
AC for FR-005 (회원가입):
□ 이메일 형식 검증 (RFC 5322)
□ 비밀번호 최소 8자, 영문+숫자+특수문자 조합
□ 중복 이메일 실시간 검증 (입력 멈춤 후 500ms)
□ 약관 동의 체크 없이 가입 버튼 비활성화
□ 가입 완료 시 인증 이메일 5분 이내 발송
```

### 경계값 테이블

숫자/범위가 관련된 AC에 적합.

```
AC for FR-012 (파일 업로드):
| 입력 | 기대 결과 |
|------|----------|
| 0 byte 파일 | "빈 파일은 업로드할 수 없습니다" 에러 |
| 1 byte ~ 10MB | 정상 업로드 + 썸네일 생성 |
| 10MB 초과 | "10MB 이하 파일만 업로드 가능합니다" 에러 |
| 허용 확장자 (jpg/png/pdf) | 정상 업로드 |
| 비허용 확장자 (exe/sh) | "허용되지 않는 파일 형식입니다" 에러 |
```

## 4. NFR AC 패턴 (비기능)

### 성능

```
NFR-P-001: 메인 페이지 LCP ≤ 2.5s (3G Fast 조건, Lighthouse 기준)
NFR-P-002: API 응답 시간 p95 ≤ 500ms (동시 접속 100명 기준)
NFR-P-003: JS 번들 크기 ≤ 300KB (gzipped)
```

### 접근성

```
NFR-A-001: WCAG 2.1 AA 전 항목 준수 (axe-core 위반 0건)
NFR-A-002: 키보드 전용 네비게이션으로 모든 기능 사용 가능
NFR-A-003: 스크린 리더(NVDA/VoiceOver)로 모든 콘텐츠 접근 가능
```

### 보안

```
NFR-S-001: OWASP Top 10 취약점 0건 (ZAP 스캔 기준)
NFR-S-002: 모든 외부 통신 TLS 1.2+ 암호화
NFR-S-003: 사용자 입력 전수 서버 사이드 검증 (XSS/SQLi 방어)
```

### 호환성

```
NFR-C-001: Chrome/Safari/Firefox/Edge 최신 2개 버전 정상 동작
NFR-C-002: iOS 15+, Android 12+ 모바일 브라우저 정상 동작
NFR-C-003: 해상도 360px~2560px 반응형 레이아웃 정상 표시
```

## 5. SOW 경계 AC 패턴

범위 내/외를 명확히 하는 AC.

```
FR-020 범위 경계:
- 포함: 상품 목록 페이지 UI + 필터링 (가격/카테고리/정렬)
- 포함: 상품 상세 페이지 UI + 이미지 갤러리
- 제외: 결제 시스템 (기존 PG 연동 유지)
- 제외: 재고 관리 시스템 (WMS 연동 별도 프로젝트)
- 회색 영역: 상품 리뷰 기능 → 고객 확인 후 포함 여부 결정
```
