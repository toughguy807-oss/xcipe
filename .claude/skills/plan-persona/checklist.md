## 구조 완전성
- [ ] 분석 개요 (분석일, 프로젝트, 데이터 소스, 페르소나 수) 명시
- [ ] 페르소나 요약 테이블 (이름/유형/연령대/역할/JTBD 요약/우선순위) 존재
- [ ] 페르소나 상세 P-### 전수 작성 (프로필/JTBD/페인/기대/인사이트/서비스 적합도)
- [ ] Primary Persona 1명 명시적 선정
- [ ] Secondary Persona 1~2명 선정
- [ ] 설계 권고 섹션 (Primary 중심 원칙, 갈등 시 우선순위, 기능 매핑) 존재

## 품질 기준
- [ ] 페르소나 3~5개 — 2개 이하면 세분화 부족, 6개 이상이면 과도
- [ ] JTBD 구체성 — "편리하게 사용" (X) / "출퇴근 시 5분 안에 예약 완료" (O)
- [ ] 페인 포인트/기대 효과 균형 — 페인만 있고 기대가 없으면 불완전
- [ ] 비예상 인사이트 전수 — 모든 페르소나에 1건 이상 필수
- [ ] Primary 선정 근거 — 왜 이 페르소나가 핵심 타겟인지 명시
- [ ] 행동/동기 기반 구분 — 인구통계만으로 구분하면 실패
- [ ] 페르소나 실재성 — 데이터 근거 없는 가상 인물은 위험
- [ ] 페르소나 간 구분 뚜렷함 — 2개 페르소나가 사실상 동일하면 합쳐야 함
- [ ] Primary 선정과 비즈니스 일치 — 사용자 수가 많아도 수익 기여가 낮으면 재검토

## 금지 사항
- [ ] "~것 같다" 등 추측성 표현 사용 금지
- [ ] 인구통계만으로 페르소나 구분 금지 — 행동/동기 기준 필수
- [ ] 비예상 인사이트 누락 금지 — 모든 페르소나에 필수
- [ ] JTBD 추상적 기술 금지 — 구체적 행동/맥락/빈도 포함
- [ ] 페인 포인트만 기술하고 기대 효과 생략 금지
- [ ] Primary Persona 선정 근거 미제시 금지
