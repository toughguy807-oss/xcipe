# 디자인 지식 수집 및 활용 정책

## 1. 목적

AI 디자인 산출물의 품질을 높이기 위한 **디자인 지식 베이스(Design Knowledge Base)** 구축 및 활용 정책.
AI는 "감성"이 아닌 "규칙"으로 디자인한다. 좋은 디자인에서 규칙을 추출하여 코드화하면 근거 기반 디자인이 가능하다.

### AI 디자인 능력의 현실적 한계

| 가능 | 불가능 |
|------|--------|
| 구조 분석 (DOM, 그리드, 섹션 순서) | 미적 판단 ("이게 더 예쁘다") |
| 패턴 빈도 통계 ("10개 중 8개가 이 방식") | 감성적 판단 ("브랜드에 맞는 톤") |
| 규칙 기반 실행 (토큰, 간격, 컬러값) | 규칙 충돌 시 우선순위 판단 |
| CSS 수치 추출 (px, rem, hex) | 문화적 맥락 판단 |

### 달성 가능 수준

| 접근 방식 | 디자이너 대비 | 설명 |
|-----------|-------------|------|
| AI 자체 판단 | ~40% | "그럴듯한" generic 결과물 |
| 동종 벤치마킹만 | ~55% | 구조는 맞지만 디테일 부족 |
| **규칙 코드화 + 다업종 레퍼런스** | **~75%** | 규칙 내에서 정확한 실행 |
| 디자이너 작업 | 100% | 규칙 + 감각 + 맥락 판단 |

## 2. 디자인 지식 베이스 구조

### 파일 구조

```
output/design/
├── DESIGN_KNOWLEDGE_{프로젝트코드}.md   ← AI 참조용 (텍스트 규칙)
└── ref/                                  ← 사람 검증용 (스크린샷)
    ├── {사이트명}_hero.png
    ├── {사이트명}_card.png
    └── ...
```

### DESIGN_KNOWLEDGE MD 포맷

```markdown
# Design Knowledge Base — {프로젝트명}

## 0. 프로젝트 디자인 방향
- 톤: {에디토리얼 / 포털 / 미니멀 / 매거진 등}
- 타겟: {주요 사용자 및 기대 경험}
- 키워드: {3~5개}

## 1. 레이아웃 패턴 (섹션별)
### {섹션명} (예: Hero)
- 채택 패턴: {구체적 설명}
- 근거: {레퍼런스 사이트명} (ref/{파일명})
- 미채택 대안: {이유와 함께}

## 2. 디자인 토큰
### Color
- Primary: #{hex} — 근거: {왜 이 색인지}
- Secondary: #{hex}
- Accent: #{hex}
- Background: #{hex}, #{hex} 교차
- Text: #{hex}

### Typography
- Font Family: {폰트명}
- Scale: {H1}px / {H2}px / {H3}px / {Body}px / {Caption}px
- Line Height: {값}
- 근거: {레퍼런스}

### Spacing
- 섹션간: {값}px — 근거: {레퍼런스}
- 카드간: {값}px
- 내부 패딩: {값}px
- Base Unit: {값}px

## 3. 컴포넌트 규칙
### Card
- 이미지 비율: {비율}
- 텍스트 줄 제한: {숫자}줄
- 호버 효과: {설명}
- 근거: {레퍼런스}

### Navigation
- 데스크톱: {설명}
- 태블릿: {설명}
- 모바일: {설명}

## 4. 이미지 가이드
- 스타일: {인물 포함 / 풍경 / 아이콘 등}
- 비율: {16:9 / 3:2 / 1:1 등}
- 톤: {밝은 자연광 / 야경 / 컬러풀 등}
- 출처: Unsplash 검색 키워드 목록

## 5. Do's & Don'ts
### DO
- {구체적 규칙 + 근거}

### DON'T
- {구체적 규칙 + 근거}
```

## 3. 레퍼런스 수집 정책

### 수집 카테고리 (다양성 필수)

동종 업계만 보면 업계 평균이 나온다. **이종 업계에서 영감을 가져와야 차별화**된다.

| 카테고리 | 목적 | 예시 |
|----------|------|------|
| **동종** | 구조/기능 패턴 | 같은 업종의 대표 사이트 3~5개 |
| **이종 - 프리미엄** | 여백/타이포/이미지 활용 | Apple, Aesop, Diptyque |
| **이종 - 에디토리얼** | 콘텐츠 톤/레이아웃 | Kinfolk, Cereal, Monocle |
| **이종 - 미디어** | 콘텐츠 큐레이션 | Netflix, Spotify |
| **트렌드** | 시대 감각 | Awwwards, GDWEB 최신 수상작 |

### 수집 방법 (우선순위 순)

1. **Chrome DevTools 전체 페이지 캡처** (권장)
   - Chrome > 개발자도구(F12) > Ctrl+Shift+P > "Capture full size screenshot"
   - 스크롤 포함 전체 페이지를 한 장으로 캡처
   - Playwright 대비 안정적, 렌더링 문제 없음

2. **섹션별 분할 캡처** (전체 캡처 실패 시)
   - 페이지를 구성 단(Hero, 콘텐츠, Footer 등)으로 나눠 캡처
   - 파일명: `{사이트명}_{섹션}.png` (예: `visitseoul_hero.png`)
   - 각 섹션의 레이아웃/비율/여백을 개별 분석

3. **Playwright 자동 캡처** (대량 수집 시)
   - `browser_take_screenshot` 도구로 자동 수집
   - 일부 사이트에서 렌더링 이슈 발생 가능 → 1번/2번으로 폴백
   - Playwright 스냅샷(DOM): 접근성 트리 구조 캡처

4. **저장 위치** — `output/design/ref/{사이트명}_{섹션}.png`

### 분석 원칙

- 스크린샷 분석은 **1회만** 수행 (토큰 절약)
- 분석 결과는 **텍스트 규칙으로 변환**하여 DESIGN_KNOWLEDGE MD에 기록
- 이후 세션에서는 텍스트만 참조, 이미지 재분석 불필요
- 스크린샷은 **사용자 검증용으로만** 보관

### 사용자 태깅 프로세스

AI가 1차 분석 후, 사용자에게 판단을 요청:

```
[Hero 섹션 분석 결과]

A. visitseoul — 풀블리드 비디오, 중앙 타이틀
B. visitjeju — 슬라이드 이미지, 좌측 카피
C. apple.com — 단일 포커스, 극단적 여백

→ 어떤 방향이 이 프로젝트에 맞습니까? (A/B/C/조합)
```

사용자 판단을 규칙으로 코드화하여 MD에 기록한다.

## 4. 적용 워크플로우

### ops 파이프라인 내 위치

```
Phase 2: 기획 (ops-planner)
    ↓
Phase 2.5: 디자인 지식 수집 ← 신규 단계
    ├── 레퍼런스 사이트 선정 (사용자 확인)
    ├── Playwright 스크린샷 수집
    ├── AI 1차 분석 + 규칙 추출
    ├── 사용자 태깅 (좋다/나쁘다/채택/미채택)
    └── DESIGN_KNOWLEDGE_{코드}.md 생성
    ↓
Phase 3: 디자인 (ops-designer)  ← DESIGN_KNOWLEDGE 참조
    ↓
Phase 4: 퍼블리싱 (ops-publisher)  ← DESIGN_KNOWLEDGE 참조
```

### 디자인 지식이 없는 경우

DESIGN_KNOWLEDGE 파일이 없으면 ops-designer는 기존 방식(AI 자체 판단)으로 작동한다.
파일이 있으면 자동으로 규칙 기반 디자인으로 전환한다.

### 이미지 활용 정책

- 목업/와이어프레임에 **실제 이미지** 사용 (placeholder 금지)
- Unsplash 등 무료 소스에서 프로젝트 성격에 맞는 이미지 활용
- DESIGN_KNOWLEDGE의 "이미지 가이드" 섹션에 검색 키워드 명시
- 최종 퍼블리싱 시에도 Unsplash 이미지로 시안 효과 유지

## 5. 품질 기준

### 디자인 지식 베이스 검증 체크리스트

- [ ] 모든 규칙에 근거(레퍼런스 사이트명 + 스크린샷 경로)가 있는가
- [ ] 사용자의 채택/미채택 판단이 기록되어 있는가
- [ ] 디자인 토큰이 CSS 변수로 바로 사용 가능한 수준인가
- [ ] Do's & Don'ts가 구체적인가 (추상적 표현 금지)
- [ ] 동종 + 이종 레퍼런스가 최소 2개 카테고리 이상 포함되어 있는가

### 갱신 정책

- 프로젝트별 1회 생성, 디자인 방향 변경 시 갱신
- 갱신 시 변경 이력 섹션에 사유 기록
- 다른 프로젝트에서 재사용 시 프로젝트 디자인 방향(섹션 0)만 교체

## 6. 디자인 검수 체크리스트 (100점 만점)

퍼블리싱 완료 후 ops-publisher가 아래 기준으로 디자인 QA를 수행한다.

### 채점 기준

| 카테고리 | 배점 | 세부 항목 |
|----------|------|-----------|
| **브랜드 일관성** | 20점 | 컬러 팔레트 준수(8), 로고 사용 규칙(4), 전체 톤 일관성(8) |
| **레이아웃 정확도** | 20점 | 그리드 준수(6), 여백 일관성(6), 정렬(4), 섹션 순서(4) |
| **타이포그래피** | 15점 | 폰트 패밀리/사이즈(6), 웨이트 위계(4), 행간/자간(5) |
| **반응형** | 15점 | 4개 BP 동작(8), 이미지 리사이즈(4), 메뉴 전환(3) |
| **코드 품질** | 10점 | 시맨틱 HTML(3), BEM 일관성(3), CSS 변수(2), 중첩 4단계↓(2) |
| **이미지 품질** | 15점 | placeholder 없음(5), 해상도 적절(3), alt 태그(3), 톤 일관성(4) |
| **기획 적합성** | 5점 | REQ/FN 반영도(3), IA 구조 일치(2) |

### 판정 기준

| 점수 | 판정 | 의미 |
|------|------|------|
| 90~100 | EXCELLENT | 바로 배포 가능 |
| 80~89 | GOOD | 경미한 수정 후 배포 |
| 70~79 | FAIR | 주요 수정 필요 |
| 70 미만 | POOR | 재작업 필요 |

### 이미지 검증

- 모든 `<img>`, `background-image` URL에 HTTP HEAD 요청
- 응답 코드 200 + content-type이 image/* 여야 정상
- `placeholder`, `via.placeholder.com`, 빈 src 등은 자동 감지
- 자동화: `node scripts/check_images.js output/publish/*.html`

## 7. 자동화 스크립트

### check_images.js — 이미지 URL 검증

```
사용법: node scripts/check_images.js <HTML파일경로> [HTML파일경로2] ...
```

- HTML에서 모든 이미지 URL 추출 (img src, srcset, background-image)
- HTTP HEAD 요청으로 유효성 검증 (타임아웃 10초)
- 결과를 JSON으로 stdout 출력, 요약은 stderr
- 깨진 이미지 1건 이상 시 exit code 1 반환

### download_gdweb.js — GDWEB 벤치마킹 이미지 다운로드

```
사용법: node scripts/download_gdweb.js [이름:str_no] [이름:str_no] ...
예시: node scripts/download_gdweb.js 강진군문화관광재단:25856
```

- GDWEB 수상작 전체 페이지 스크린샷을 자동 다운로드
- Puppeteer 기반, 시스템 Chrome 자동 탐지
- 저장 위치: `output/리서치/벤치마킹_{이름}.jpg`
- 벤치마킹 레퍼런스 수집 Phase 2.5에서 사용
