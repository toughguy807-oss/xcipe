# 산출물 품질 기준

## 기본 원칙
- 기존 산출물이 입력으로 주어지면, 그 품질을 기준선으로 유지하고 부족한 것만 보강
- 출력 전 자체 점검: "이게 입력보다 나은가?"
- 심플 우선. 불필요한 시각 장식(다크테마, 뱃지, 탭, 인터랙션)은 가독성을 해침
- 도식화가 필요하면 Canva/미리캔버스 대시보드 벤치마킹. 임의 CSS 금지

## 구조적 완결성
- MECE: 빠진 것 없고(Exhaustive), 중복 없는(Exclusive) 목차 구성
- 모든 제안은 Fact(현상) → Insight(해석) → Action(제안) 구조

## 커뮤니케이션 원칙
- 두괄식: 결론부터 말한다
- 단정적 어조: "~것 같습니다" 금지 → "~입니다" / "~로 판단됩니다"
- 정량적 근거: "성능 향상" (X) → "로딩 1.2초 단축" (O)
- 불필요한 서두/겸양/미사여구 금지. 바로 본론 진입

## 커맨드에 반드시 포함할 것
- **품질 기준 섹션**: "칸을 채워라"가 아니라 "이 수준으로 만들어라"
- **고품질 예시**: 구조만 있고 예시가 없으면 출력이 generic해짐
- **변경 사유 필수**: 모든 의사결정에 "왜"가 달려야 보고서로 쓸 수 있음

## 도메인별 실패 정의 매트릭스 (2026-04-30 추가)

**이유**: Addy Osmani "성공은 조용하고, 실패는 시끄럽다" — 좋은 하네스는 실패를 증폭한다. 실패 정의가 없으면 시스템은 침묵하고 학습하지 못한다. 시각/성능 도메인은 실패가 명확(LCP/CLS/픽셀 매칭)하나, 기획/요구사항 도메인은 모호 → 도메인 불균형.

**원칙**: 모든 도메인은 "이것이 발견되면 PASS 불가"인 **실패 시그널**을 정량/정성적으로 명시한다. reviewer/Self-Check는 본 매트릭스를 기준으로 채점한다.

| 도메인 | 정량 실패 시그널 | 정성 실패 시그널 |
|--------|---------------|---------------|
| **plan-qst** | 필수 카테고리 누락 (비즈니스/KPI/팀/예산/일정 中) ≥1개 | "고객이 답변 못할 질문" 1개 이상, 선택지 없는 개방형 남발 |
| **plan-req** | FR ID 중복/누락, AC 미작성 FR ≥1개, NFR 측정 기준 없는 항목 ≥1개 | 비즈니스 규칙과 FR 충돌, 추적성 단절 |
| **plan-fn** | FR↔FN 매핑 단절 ≥1건, 검증 3단계(정상/예외/에러) 미작성 ≥1건 | "어떻게 동작하는지 추측해야" 하는 서술, 의존관계 누락 |
| **plan-ia** | 사이트맵-페이지인벤토리-URL 3종 단절, 고아 페이지 ≥1개 | 메뉴 명명 불일치, 사용자 흐름 끊김 |
| **plan-wbs** | 크리티컬 패스 미식별, 리소스 미배정 작업 ≥1개 | 일정 산정 근거 없음, 의존관계 모순 |
| **design-knowledge** | 7대 토큰 카테고리 누락 ≥1개, 컬러 대비비 4.5:1 미달 | 브랜드 시트 미반영, 토큰 명명 불일치 |
| **design-layout** | 3 breakpoint(M/T/D) 미작성 ≥1개 페이지 | 그리드 일관성 없음, 와이어프레임 빈 박스 |
| **design-ui** | 컴포넌트 상태(default/hover/active/disabled) 미작성 ≥1건 | UI↔FN 불일치, 인터랙션 명세 모호 |
| **publish-markup** | HTML validation 에러 ≥1건, BEM 명명 위반 ≥1건 | ARIA/시맨틱 태그 부재, 접근성 트리 깨짐 |
| **publish-style** | Never-Rules 위반 ≥1건 (publish-patterns.md), CLS > 0.1 | Iron Rules 5축 미달, 토큰 미사용 하드코딩 |
| **publish-interaction** | 빈 산출물(JS 0줄), Hero invisible, console error ≥1건 | 인터랙션 명세 미구현, 모션 끊김 |
| **dev-spec** | API 엔드포인트 누락 ≥1건, ERD-FN 단절 | 시퀀스 다이어그램 모호, 에러 처리 매트릭스 누락 |
| **dev-component** | 정적 HTML 잔존(미변환), 상태 바인딩 누락 | 훅 미작성, 컴포넌트 분리 부적절 |
| **qa-functional** | TC-FN 추적성 단절 ≥1건, Critical 미해결 ≥1건 | 정상/예외/에러 3단계 미커버 |
| **qa-accessibility** | WCAG 2.1 AA 위반 ≥1건 (자동 검출 도구 기준) | 키보드 트랩, 스크린리더 역할 누락 |
| **qa-performance** | LCP > 2.5s, CLS > 0.1, FCP > 1.8s | 리소스 최적화 미적용, NFR 미달 |
| **qa-security** | OWASP Top 10 위반 ≥1건, MCP 인젝션 취약점 ≥1건 | 시크릿 하드코딩, 권한 게이트 누락 |
| **ops-intake** | 프로젝트 식별 실패, 티켓 분류 모호 | Notion 동기화 실패 무시 |

**reviewer 채점 규칙**:
- 정량 시그널 1건 발견 = -10점
- 정성 시그널 1건 발견 = -5점
- 100점 만점에서 차감 후 도메인별 게이트 점수와 비교

**시그널 침묵 차단**: reviewer/Self-Check는 "발견 0건" 결과를 출력할 때 **반드시 어떤 시그널을 검사했는지 열거**해야 한다. 검사 항목 명시 없는 PASS는 무효.

## 신뢰도 정량 자가 보고 (2026-04-30 추가)

**이유**: Debs O'Brien E2E 테스트 사례 4원칙 中 "신뢰도 명시적으로 확인" — "확신도가 몇 점인가요? 왜 7일까요?". 모델은 그럴듯한 합리화를 만들기 쉬움. 신뢰도 정량화는 **합리화를 외부화**하는 가장 단순하고 효과적인 게이트.

**규칙**: reviewer/Self-Check가 PASS를 보고할 때 다음 3개 정보를 **반드시 함께 출력**한다.

| 항목 | 형식 | 예시 |
|------|------|------|
| 신뢰도 | 1~10 정수 | `신뢰도: 8/10` |
| 근거 | 1줄 (검사 시그널 또는 검증 도구 결과 인용) | `근거: FR-FN 매핑 27/27 100%, NFR 측정 기준 6/6 명시` |
| 미확인 영역 | 1줄 (검사 안 한 영역 + 사유) | `미확인: 비즈니스 규칙↔FR 충돌 (수동 검증 필요)` |

**임계 규칙**:
- 신뢰도 ≤ 6 → **자동 reviewer 호출** (Sonnet/high) 또는 사용자 에스컬레이션
- 신뢰도 7 → 진행 가능, 단 미확인 영역을 _handoff.md에 기록
- 신뢰도 ≥ 8 → 정상 진행

**금지**:
- 근거 없는 신뢰도 출력 ("8/10 정도 됩니다") → 무효
- "거의 확실" / "대체로 OK" 같은 정성 표현 → 합리화로 간주
- 미확인 영역 "없음" 자동 출력 → 검사 항목을 열거하지 않은 것과 동일
