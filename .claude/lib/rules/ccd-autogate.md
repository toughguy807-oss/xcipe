# CCD 자동 게이트 프로토콜

CCD(Claude Code Driven) 모드에서 오케스트레이터의 Gate 선택지를 자동 처리하는 규칙.

## 적용 조건

CCD 모드가 활성화된 프로젝트에서만 적용한다.
활성화 기준: 사용자가 "CCD", "자동 진행", "PM 개입 없이" 중 하나를 명시.

## Gate 자동 결정 규칙

### 1. Self-Check 결과 기반

| Self-Check 결과 | Gate 결정 |
|----------------|----------|
| PASS (전 항목) | **A) 다음 진행** — 자동 통과 |
| FAIL 1~2건 (경미) | **수정 후 A)** — 자동 수정 → 재검증 → 통과 |
| FAIL 3건 이상 | **에스컬레이션** — 사용자에게 보고 |

### 2. DA 챌린지 결과 기반

| DA 결과 | Gate 결정 |
|---------|----------|
| 전 항목 PM-OK | **A) 진행** — 단, anti-rationalization Red Flag #2 점검 |
| PM-WARN 존재 | **A) 진행** — WARN 내용을 _context.md에 기록 후 진행 |
| PM-BLOCK 존재 | **에스컬레이션** — 사용자에게 BLOCK 사유 보고 후 대기 |

### 3. reviewer 채점 기반 (도메인별 임계값)

도메인별 reviewer 100점 채점 결과에 따른 자동 결정. 이 표는 design/publish/qa-orchestrator에서 공통 참조한다 (각 orch에서 표 복제 금지).

| 도메인 | PASS 통과 | 이터레이션 (최대 3회) | 에스컬레이션 |
|--------|----------|--------------------|-----------|
| **planning** | 90점+ | 75~89점 | < 75점 |
| **design** | 90점+ | 75~89점 | < 75점 |
| **publish** | 80점+ | 60~79점 | < 60점 |
| **qa** | 80점+ | 60~79점 | < 60점 |

> 임계값 차이 사유: planning/design은 시각·전략 산출물(주관 평가 비중↑)로 높은 컷오프, publish/qa는 정량 측정 비중↑로 80점 컷오프.

이터레이션 분기: P0/P1 이슈 0건까지 루프, 초과 시 에스컬레이션.

### 4. 선택적 단계 자동 판단

| 조건 | 결정 |
|------|------|
| 프로젝트 복잡도 높음 (FN 15건+) | IA, WBS 포함 |
| 프로젝트 복잡도 보통 (FN 8~14건) | IA 포함, WBS 생략 |
| 프로젝트 복잡도 낮음 (FN 7건 이하) | IA, WBS 모두 생략 |
| 디자인 레퍼런스 없음 | design-benchmark 포함 |
| Figma 시안 존재 + MCP 활성 | design-benchmark **생략** → Step 2A (Figma→Code) 직행 |
| Figma 시안 존재 + MCP 비활성 | design-benchmark 포함 (Figma를 육안 참조하되 벤치마킹→HTML 경로) |
| Figma MCP 토큰 `<FIGMA_ACCESS_TOKEN>` (미설정) | MCP 비활성과 동일 처리 |
| Figma URL 없음 | design-benchmark **필수** 포함 (벤치마킹→HTML 경로) |

## PM Direction 자동 추출

CCD 모드에서 PM Direction Gate를 자동 통과하려면, 프롬프트에서 아래 정보를 추출한다:

| 필요 정보 | 추출 방법 | Fallback |
|----------|----------|---------|
| 프로젝트명 | 프롬프트의 첫 명사구 | "Untitled" |
| 업종 | 키워드 매칭 (카페/ERP/쇼핑몰 등) | "일반" |
| 규모 | 숫자 추출 (직원 수, 페이지 수) | "소규모" |
| 핵심 기능 | 동사+명사 패턴 추출 | 프롬프트 전문 |
| 플랫폼 | 키워드 (웹/앱/SaaS/VR) | "웹" |

추출 실패 시: 해당 필드를 `[자동 추정]`으로 표기하고 진행. 에스컬레이션하지 않는다.

### 5. 비가역적 외부 작업 — 사람 승인 필수

CCD 모드라도 아래 작업은 **자동 실행하지 않는다**. 반드시 사용자에게 확인 후 진행한다.

| 작업 유형 | 예시 | 사유 |
|----------|------|------|
| **배포** | 프로덕션/스테이징 배포, Docker push, CI/CD 트리거 | 비가역적 외부 영향 |
| **머지** | PR 머지, main/master 브랜치 변경, force push | 타인 작업 영향 |
| **외부 전송** | 이메일 발송, Slack 메시지, API 호출(POST/PUT/DELETE) | 되돌릴 수 없는 커뮤니케이션 |
| **데이터 삭제** | DB 레코드 삭제, 파일 시스템 rm -rf, 브랜치 삭제 | 비가역적 데이터 손실 |
| **권한 변경** | IAM/RBAC 설정, 환경변수 변경, 시크릿 업데이트 | 보안 영향 |

> CCD 자동 진행 원칙의 예외: 이 5가지는 "에스컬레이션"이 아닌 **"하드 게이트"**다. 사용자 승인 없이 절대 실행하지 않는다.

### 6. DKB DQG (Design Quality Gate) — CCD 자동 통과 조건

CCD 모드에서 디자인 시각 산출물 단계는 DQG 통과를 자동 판정하되, **3회 재생성 후 미달은 사용자 게이트**를 유지한다.

| DQG 결과 | CCD 처리 |
|---------|---------|
| 145+/180 PASS + DA PM-OK | **A) 진행** — 최고 점수 시안 자동 선정 |
| 130~144 + DA PM-WARN | **자동 재생성 1회** → 145+ PASS 재시도 |
| <130 또는 DA PM-BLOCK | **자동 재생성 (최대 3회)** — references 변경 / 약점 축 강화 |
| 3회 재생성 후 미달 | ❌ **에스컬레이션** — 자동 강행 금지 (`lib/rules/dkb-policy.md` §6) |

**DKB references 매칭 0건 시**: CCD 모드에서도 자동 진행 금지. dkb-analyze 수동 트리거 권고 후 사용자 게이트.

**근거**: 다회 재시도 사고 케이스가 CCD 자동 진행 + LLM 자가 채점 합리화로 발생. DKB 단계는 "외부 Vision 검증자" 강제 + 사용자 게이트 유지 (`lib/rules/anti-rationalization.md`).

## 에스컬레이션 시 출력 형식

```
═══════════════════════════════════
[에스컬레이션] {단계명}
═══════════════════════════════════
사유: {PM-BLOCK 또는 FAIL 3건+ 또는 60점 미만}
상세: {구체적 문제 내용}
───────────────────────────────────
선택지:
A) 문제 해소 후 재시도
B) 무시하고 진행 (품질 리스크 수용)
C) 파이프라인 중단
═══════════════════════════════════
```
